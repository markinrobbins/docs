## O que e um guard

Um guard e uma classe que responde uma unica pergunta antes de qualquer comando ou listener rodar: essa execucao deve continuar? Se o guard diz sim, o handler segue normalmente. Se o guard diz nao, o framework para a execucao imediatamente e o metodo do handler nunca roda. Esse e o conceito central por tras dos guards no Spraxium, e tudo nessa categoria e construido em cima dessa decisao simples de permitir ou negar.

Na pratica, isso significa que voce nao precisa escrever checagens de permissao, validacoes de role ou logica de rate limit dentro de cada metodo handler. Em vez disso, voce escreve essa logica uma unica vez em uma classe guard e aplica onde for necessario. O guard roda automaticamente antes do handler, e o handler lida apenas com logica de negocio porque a fronteira de autorizacao ja aconteceu. Isso mantem seus command handlers limpos e sua politica de acesso centralizada em classes dedicadas e reutilizaveis.

Guards tambem se integram com a camada de exceptions. Quando um guard nega execucao, o framework pode renderizar uma resposta pro usuario atraves de exception layouts sem que o handler precise saber sobre isso. Essa separacao e o que faz dos guards uma fronteira arquitetural de verdade, e nao apenas uma funcao utilitaria que voce chama manualmente no inicio de cada handler.

## A interface SpraxiumGuard

Todo guard implementa a interface *SpraxiumGuard*, que exige um unico metodo chamado *canActivate*. Esse metodo recebe o *ExecutionContext* atual e deve retornar um boolean ou uma promise que resolve pra boolean. Se o metodo retorna true, a execucao continua pro proximo guard ou pro handler. Se retorna false, a execucao para e o framework trata como uma negacao.

```typescript filename="guild-only.guard.ts"
import { Guard } from '@spraxium/common';
import type { ExecutionContext, SpraxiumGuard } from '@spraxium/common';

@Guard()
export class GuildOnlyGuard implements SpraxiumGuard {
  canActivate(ctx: ExecutionContext): boolean {
    return ctx.getGuildId() !== null;
  }
}
```

O metodo *canActivate* suporta retornos sincronos e assincronos. Se seu guard precisa checar um banco de dados, chamar uma API externa ou fazer qualquer operacao async pra decidir se a execucao deve continuar, voce pode retornar uma *Promise* e o framework vai aguardar o resultado antes de tomar a decisao. Essa flexibilidade significa que guards funcionam igualmente bem pra checagens instantaneas como verificar contexto de guild e pra checagens mais pesadas como consultar um banco de dados de assinaturas.

## O ExecutionContext

O *ExecutionContext* e o objeto passado pro metodo *canActivate* de todo guard. Ele encapsula uma interaction ou message do Discord e fornece uma API unificada pra que seu guard nao precise saber se o comando foi acionado por um slash command ou um prefix command. Cada metodo do contexto extrai uma informacao especifica sobre quem acionou a execucao e onde aconteceu.

Voce pode estar se perguntando por que *getUserId* e *getUser* existem separadamente. O motivo e performance: se seu guard so precisa comparar o ID do usuario contra uma lista de owner IDs, chamar *getUserId* evita resolver o objeto User completo. Se seu guard precisa de mais informacoes como a data de criacao da conta, *getUser* te da o objeto completo. Use o que fizer sentido pra checagem especifica do seu guard.

## O pipeline de execucao

Guards nao rodam em ordem aleatoria. O Spraxium avalia em tres camadas, sempre na mesma sequencia: guards globais primeiro, depois guards de classe, depois guards de metodo. Se todos os guards em todas as tres camadas retornam true, o metodo handler executa. Se qualquer guard retorna false ou lanca uma exception em qualquer ponto da cadeia, a execucao para imediatamente e todos os guards restantes sao pulados.

<Steps>
  <Step title="Guards globais rodam primeiro">
    Sao registrados durante o bootstrap da aplicacao com *useGlobalGuards* na instancia *SpraxiumApplication*. Eles se aplicam a todo comando e todo listener com guard na aplicacao inteira, o que os torna o lugar certo pra regras amplas como bloquear contas de bot, restringir acesso durante manutencao ou aplicar rate limit global.
  </Step>
  <Step title="Guards de classe rodam em segundo">
    Sao aplicados com *UseGuards* na classe do handler ou listener. Eles se aplicam a todo metodo dentro daquela classe, o que e util quando todos os comandos de um modulo compartilham o mesmo requisito de acesso. Um exemplo tipico e aplicar um guard de role na classe pra que todo subcommand de um handler de moderacao exija a role Moderator.
  </Step>
  <Step title="Guards de metodo rodam por ultimo">
    Sao aplicados com *UseGuards* diretamente em um metodo especifico. Eles se aplicam apenas aquele metodo, o que permite adicionar requisitos mais rigorosos pra operacoes sensiveis como banir usuarios ou resetar dados sem afetar o resto dos metodos da classe.
  </Step>
  <Step title="Handler executa">
    Somente depois que todo guard em toda camada retornou true o framework chama o metodo handler de fato. Se seu codigo chega ao handler, voce pode ter certeza de que todas as politicas de acesso configuradas ja foram satisfeitas pro contexto de execucao atual.
  </Step>
</Steps>

Esse pipeline usa avaliacao short-circuit. No momento em que qualquer guard nega a execucao, o framework para de avaliar e vai direto pro tratamento de negacao. Isso e importante pra performance porque voce deve colocar checagens baratas antes das caras sempre que possivel. Por exemplo, uma simples checagem de guild-only que le uma propriedade deve vir antes de uma consulta ao banco que verifica status de assinatura premium.

<Callout tone="info" title="Guard versus logica de negocio">
  Um guard responde "isso pode rodar?" e nada mais. O handler responde "o que deve acontecer?". Mantenha essas responsabilidades separadas pra que guards fiquem pequenos, reutilizaveis e faceis de testar independentemente.
</Callout>
