## Qué es un prefix command

Un prefix command es un par de clases que reacciona a un mensaje de texto que comienza con un carácter configurable como `!` o `?`. Cuando un usuario envía `!roll 20`, el framework detecta el prefijo, extrae el nombre del comando, parsea los argumentos, ejecuta guards y despacha el mensaje al handler correcto. Este es el modelo tradicional de comandos de bots Discord que precede a los slash commands, y sigue siendo útil para workflows rápidos de operación, herramientas internas de moderación y bots donde los usuarios ya conocen los atajos.

La diferencia clave entre prefix commands y slash commands es lo que los activa. Los slash commands se activan por el sistema de interacciones nativo de Discord con autocomplete y validación. Los prefix commands se activan por mensajes de texto crudos, lo que significa que el framework debe manejar el parsing, la coerción de argumentos y el reporte de errores por su cuenta. Algunos bots necesitan ambos, y Spraxium permite registrar prefix y slash commands en el mismo módulo sin conflictos.

Los prefix commands en Spraxium no son un legado olvidado. Son clases completas manejadas por el contenedor de inyección de dependencias, registradas en módulos y conectadas al client de Discord automáticamente durante el bootstrap. Esto significa que tus handlers pueden recibir servicios inyectados, participar en el pipeline de guards, beneficiarse del seguimiento de cooldown y usar el mismo manejo de excepciones que los slash commands.

## Un ejemplo básico de comando

Aquí hay dos clases que muestran el patrón de dos clases. La primera es una definición de comando que declara el nombre, aliases y schema de argumentos a través de decorators *@PrefixArg*. La segunda es un handler que recibe los argumentos parseados y el *Message* crudo vía *@Ctx*. Juntas, cubren la estructura mínima que necesitas para crear un prefix command funcional.

<CodeTabs>
  <Tab label="Comando" language="typescript">
```typescript filename="src/modules/fun/commands/ping.command.ts"
import { PrefixCommand } from '@spraxium/common';

@PrefixCommand({
  name: 'ping',
  aliases: ['p'],
  description: 'Check bot latency.',
  category: 'Fun',
  usage: '!ping',
  cooldown: 3,
})
export class PingCommand {
  build() {}
}
```
  </Tab>
  <Tab label="Handler" language="typescript">
```typescript filename="src/modules/fun/handlers/ping.handler.ts"
import { Ctx, PrefixCommandHandler } from '@spraxium/common';
import type { Message } from 'discord.js';
import { PingCommand } from '../commands/ping.command';

@PrefixCommandHandler(PingCommand)
export class PingHandler {
  async handle(@Ctx() message: Message): Promise<void> {
    const latency = Date.now() - message.createdTimestamp;
    await message.reply(`Pong! Latency: ${latency}ms`);
  }
}
```
  </Tab>
</CodeTabs>

Nota que la clase de comando es puramente declarativa: describe cómo se llama el comando, qué aliases acepta y qué cooldown impone. La clase handler contiene la lógica de ejecución y recibe dependencias a través de decorators. Para comandos con argumentos, agregas decorators *@PrefixArg* en el método *build* del comando y decorators *@PrefixArg* correspondientes en los parámetros del método *handle* del handler. La siguiente página cubre schemas de argumentos en detalle.

## Registrando comandos en un módulo

Los prefix commands no se activan solo por existir en tu código fuente. Necesitas registrar la clase de comando en el array *commands* del módulo y la clase handler en el array *handlers* para que el framework sepa sobre ellos y pueda conectarlos durante el bootstrap. Sin ambos registros, tu comando será invisible para el parser o no tendrá handler para despacho.

```typescript filename="src/modules/fun/fun.module.ts"
import { Module } from '@spraxium/common';
import { PingCommand } from './commands/ping.command';
import { PingHandler } from './handlers/ping.handler';

@Module({
  commands: [PingCommand],
  handlers: [PingHandler],
})
export class FunModule {}
```

El módulo en sí debe ser importado por tu módulo raíz u otro módulo en la cadena de imports. El framework procesa módulos durante el bootstrap, resuelve los arrays de commands y handlers, instancia cada clase a través del contenedor de DI y luego los conecta al prefix dispatcher. Si un handler depende de un servicio, ese servicio debe estar disponible en el alcance de providers del módulo o exportado desde un módulo importado, igual que cualquier otra dependencia inyectable.

## Cómo el bootstrap conecta prefix commands

El proceso de bootstrap maneja prefix commands en una secuencia específica que importa para entender cuándo tu comando se activa. Primero, el framework lee los arrays *commands* y *handlers* de cada módulo. Segundo, instancia cada clase por el contenedor de DI, resolviendo dependencias de constructor. Tercero, el *PrefixRegistry* indexa todos los nombres de comando, aliases, rutas de subcomandos y schemas de argumentos para búsqueda rápida. Cuarto, el *PrefixBinder* conecta un único listener *messageCreate* al client de Discord. Finalmente, cuando un mensaje llega, el *PrefixInvoker* ejecuta el pipeline completo: detección de prefijo, tokenización, verificación de cooldown, evaluación de guards, parsing de argumentos y despacho del handler.

Esta estructura mantiene definiciones de comando y handlers en carpetas dedicadas y el archivo de módulo en la raíz del dominio. Conforme tu bot crece, puedes agregar más archivos de comando y handler sin cambiar la estructura de carpetas. Cada comando tiene exactamente un handler (o uno por subcomando), y el archivo de módulo es el único lugar que declara cuáles comandos están activos.

<Callout tone="info" title="Dos clases por comando">
  Cada prefix command requiere una clase de comando y una clase handler registradas en el mismo módulo. La clase de comando declara estructura y metadatos. La clase handler contiene lógica de ejecución. Si olvidas registrar cualquiera de las dos, el comando no funcionará y el framework no producirá un error en el arranque.
</Callout>
